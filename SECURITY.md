# Security policy

## Reporting a vulnerability

Do not disclose a suspected vulnerability in a public issue. Use the repository host's private security-advisory facility, or contact the maintainers privately through the contact method listed on the repository profile. Include affected versions, reproduction steps, impact and any suggested mitigation. Maintainers will acknowledge and triage reports as availability permits.

## Scope and limitations

The default examples are offline and use read-only or simulated tools. Exact-action approval, allowlists and prompt-injection controls reduce risk but are not a complete security guarantee. Never use tutorial policies as the sole control for consequential production systems.

Model files are not distributed with the repository. Verify the recorded checksum before loading an optional GGUF file. Never place credentials in configuration, traces, notebooks or issues.

Only the latest release candidate is actively assessed. No formal response-time or long-term support commitment is currently offered.
